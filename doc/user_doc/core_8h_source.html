658  * the constraints over the iteration domains of each computation in
659  * the function.
660  *
661  * In order to deduce bounds, Tiramisu first identifies the final consumers
662  * in the function (i.e., computations that do not have any consumer).
663  * Then, it propagates the bounds over the final consumers to their producers.
664  * The bounds of each consumer are used to deduce the bounds over its producer.
665  *
666  * To take benefit of bound inference, the user can declare computations
667  * without providing constraints on their iteration domains. For example
668  * the user can declare the following computations (the left side is the
669  * iteration domain, while the right side is the expression attached to
670  * each computation)
671  *
672  * \code
673  * {A[i] } : 0
674  * {B[i] } : 0
675  * {C[i] } : A[i] + B[i]
676  * {D[i]: 0<=i<N} : 2*C[i]
677  * \endcode
678  *

2162  /**
2163  * Set the schedule indicated by \p map.
2164  *
2165  * \p map is a string that represents a mapping from the iteration domain
2166  * to the time-space domain (the ISL format to represent maps is
2167  * documented in http://barvinok.gforge.inria.fr/barvinok.pdf in Sec 1.2.2).
2168  *
2169  * The schedule is a map from the iteration domain to a time space
2170  * domain. The same name of space should be used for both the range
2171  * and the domain of the schedule.
2172  *
2173  * In general, users can set the schedule using high level functions such
2174  * as before(), after(), tile(), compute_at(), vectorize(), split(), ...
2175  * The use of this function is only reserved for advanced users who want
2176  * a low level control of the schedule.
2177  *
2178  * Vectors in the time-space domain have the following form
2179  *
2180  * computation_name[redundancy_ID,static,dynamic,static,dynamic,static,dynamic,static,...]
2181  *
2182  * The first dimension of the vector is used to indicate the redundancy ID
2183  * (the notion of the redundancy ID is explained later).
2184  *
2185  * The following dimensions are interleaved dimensions: static, dynamic, static,
2186  * dynamic, ...
2187  * Dynamic dimensions represent the loop levels, while static dimensions are
2188  * used to order statements within a given block of statements in a given loop
2189  * level.
2190  * For example, the computations c0 and c1 in the following loop nest
2191  *
2192  * for (i=0; i<N: i++)
2193  * for (j=0; j<N; j++)
2194  * {
2195  * c0;
2196  * c1;
2197  * }
2198  *
2199  * have the following representations in the iteration domain
2200  *
2201  * {c0[i,j]: 0<=i<N and 0<=j<N}
2202  * {c1[i,j]: 0<=i<N and 0<=j<N}
2203  *
2204  * and the following representation in the time-space domain
2205  *
2206  * {c0[0,0,i,0,j,0]: 0<=i<N and 0<=j<N}
2207  * {c1[0,0,i,0,j,1]: 0<=i<N and 0<=j<N}
2208  *
2209  * The first dimension (dimension 0) in the time-space
2210  * domain (the leftmost dimension) is the redundancy ID
2211  * (in this case it is 0, the meaning of this ID will be explained later).
2212  * The second dimension (starting from the left) is a static dimension,
2213  * the third dimension is a dynamic dimension that
2214  * represents the loop level i, ..., the fifth dimension is a dynamic
2215  * dimension that represents the loop level j and the last dimension
2216  * (dimension 5) is a static dimension and allows the ordering of
2217  * c1 after c0 in the loop nest.
2218  *
2219  * To transform the previous iteration domain to the
2220  * time-space domain, the following schedule should be used:
2221  *
2222  * {c0[i,j]->c0[0,0,i,0,j,0]: 0<=i<N and 0<=j<N}
2223  * {c1[i,j]->c1[0,0,i,0,j,1]: 0<=i<N and 0<=j<N}
2224  *
2225  * The first dimension called "redundancy ID" is only meaningful if the
2226  * computation is redundant. i.e., some parts of the computation are
2227  * redundantly computed. Redundant computations are in general used to
2228  * maximize parallelism and data locality on the expense of doing some
2229  * computations redundantly.
2230  * For example, if the two computations c1(i,j) and c2(i,j) both depend
2231  * on the computation c0(i,j), instead of waiting for c0(i,j) to be
2232  * computed and then computing c1(i,j) and c2(i,j) in parallel, the thread
2233  * executing c1(i,j) can compute c0(i,j) by itself and then run c1(i,j).
2234  * The thread that computes c2(i,j) can do the same and compute c0(i,j)
2235  * by itself and then compute c2(i,j). In this case the two threads do not
2236  * need to wait. This is done at the expense of redundant computation since
2237  * c0(i,j) is computed by both threads.
2238  *
2239  * In general redundant computations are useful when tiling stencil
2240  * computations. In the context of stencils such a tiling is called
2241  * "overlapped tiling". Tiles that depend on results computed by other
2242  * tiles that run in parallel can compute the boundaries redundantly which
2243  * allows them to avoid waiting and thus can run in parallel.
2244  *
2245  * In Tiramisu, the user can indicate that a chunk of a computation
2246  * should be computed redundantly. The original computation always has a redundancy
2247  * ID equal to 0 (which means this is the original computation).
2248  * The redundant chunk has an ID that is different from 0 and that is
2249  * used to uniquely identify it.
2250  *
2251  * For example if we want to compute all of c0 three times (that is,
2252  * compute the original computation and compute two redundant computations),
2253  * we can use the following schedules:
2254  *
2255  * The schedule of the original computation: {c0[i,j]->c0[0,0,i,0,j,0]: 0<=i<N and 0<=j<N}
2256  * The schedule of the redundant computation N°1: {c0[i,j]->c0[1,0,i,0,j,0]: 0<=i<N and 0<=j<N}
2257  * The schedule of the redundant computation N°2: {c0[i,j]->c0[2,0,i,0,j,0]: 0<=i<N and 0<=j<N}
2258  *
2259  * The schedule of c0 in this case would be three maps that map c0[i,j] to
2260  * the three different redundant computations in the time-processor domain:
2261  *
2262  * {c0[i,j]->c0[0,0,i,0,j,0]: 0<=i<N and 0<=j<N;
2263  * c0[i,j]->c0[1,0,i,0,j,0]: 0<=i<N and 0<=j<N;
2264  * c0[i,j]->c0[2,0,i,0,j,0]: 0<=i<N and 0<=j<N}
2265  *
2266  * The function set_schedule() overrides any other schedule set by the high level
2267  * scheduling functions. Currently the user has to choose between using the high
2268  * level scheduling functions or using this low level set_schedule function. The user
2269  * cannot mix the use of the two in the same program because they are not compatible.
2270  */
2271  // @{
2272  void set_schedule(isl_map *map);

2854  /**
2855  * Set the schedule indicated by \p map.
2856  *
2857  * \p map is a string that represents a mapping from the iteration domain
2858  * to the time-space domain (the ISL format to represent maps is
2859  * documented in http://barvinok.gforge.inria.fr/barvinok.pdf in Sec 1.2.2).
2860  *
2861  * The schedule is a map from the iteration domain to a time space
2862  * domain. The same name of space should be used for both the range
2863  * and the domain of the schedule.
2864  *
2865  * In general, users can set the schedule using high level functions such
2866  * as before(), after(), tile(), compute_at(), vectorize(), split(), ...
2867  * The use of this function is only reserved for advanced users who want
2868  * a low level control of the schedule.
2869  *
2870  * Vectors in the time-space domain have the following form
2871  *
2872  * computation_name[redundancy_ID,static,dynamic,static,dynamic,static,dynamic,static,...]
2873  *
2874  * The first dimension of the vector is used to indicate the redundancy ID
2875  * (the notion of the redundancy ID is explained later).
2876  *
2877  * The following dimensions are interleaved dimensions: static, dynamic, static,
2878  * dynamic, ...
2879  * Dynamic dimensions represent the loop levels, while static dimensions are
2880  * used to order statements within a given block of statements in a given loop
2881  * level.
2882  * For example, the computations c0 and c1 in the following loop nest
2883  *
2884  * for (i=0; i<N: i++)
2885  * for (j=0; j<N; j++)
2886  * {
2887  * c0;
2888  * c1;
2889  * }
2890  *
2891  * have the following representations in the iteration domain
2892  *
2893  * {c0[i,j]: 0<=i<N and 0<=j<N}
2894  * {c1[i,j]: 0<=i<N and 0<=j<N}
2895  *
2896  * and the following representation in the time-space domain
2897  *
2898  * {c0[0,0,i,0,j,0]: 0<=i<N and 0<=j<N}
2899  * {c1[0,0,i,0,j,1]: 0<=i<N and 0<=j<N}
2900  *
2901  * The first dimension (dimension 0) in the time-space
2902  * domain (the leftmost dimension) is the redundancy ID
2903  * (in this case it is 0, the meaning of this ID will be explained later).
2904  * The second dimension (starting from the left) is a static dimension,
2905  * the third dimension is a dynamic dimension that
2906  * represents the loop level i, ..., the fifth dimension is a dynamic
2907  * dimension that represents the loop level j and the last dimension
2908  * (dimension 5) is a static dimension and allows the ordering of
2909  * c1 after c0 in the loop nest.
2910  *
2911  * To transform the previous iteration domain to the
2912  * time-space domain, the following schedule should be used:
2913  *
2914  * {c0[i,j]->c0[0,0,i,0,j,0]: 0<=i<N and 0<=j<N}
2915  * {c1[i,j]->c1[0,0,i,0,j,1]: 0<=i<N and 0<=j<N}
2916  *
2917  * The first dimension called "redundancy ID" is only meaningful if the
2918  * computation is redundant. i.e., some parts of the computation are
2919  * redundantly computed. Redundant computations are in general used to
2920  * maximize parallelism and data locality on the expense of doing some
2921  * computations redundantly.
2922  * For example, if the two computations c1(i,j) and c2(i,j) both depend
2923  * on the computation c0(i,j), instead of waiting for c0(i,j) to be
2924  * computed and then computing c1(i,j) and c2(i,j) in parallel, the thread
2925  * executing c1(i,j) can compute c0(i,j) by itself and then run c1(i,j).
2926  * The thread that computes c2(i,j) can do the same and compute c0(i,j)
2927  * by itself and then compute c2(i,j). In this case the two threads do not
2928  * need to wait. This is done at the expense of redundant computation since
2929  * c0(i,j) is computed by both threads.
2930  *
2931  * In general redundant computations are useful when tiling stencil
2932  * computations. In the context of stencils such a tiling is called
2933  * "overlapped tiling". Tiles that depend on results computed by other
2934  * tiles that run in parallel can compute the boundaries redundantly which
2935  * allows them to avoid waiting and thus can run in parallel.
2936  *
2937  * In Tiramisu, the user can indicate that a chunk of a computation
2938  * should be computed redundantly. The original computation always has a redundancy
2939  * ID equal to 0 (which means this is the original computation).
2940  * The redundant chunk has an ID that is different from 0 and that is
2941  * used to uniquely identify it.
2942  *
2943  * For example if we want to compute all of c0 three times (that is,
2944  * compute the original computation and compute two redundant computations),
2945  * we can use the following schedules:
2946  *
2947  * The schedule of the original computation: {c0[i,j]->c0[0,0,i,0,j,0]: 0<=i<N and 0<=j<N}
2948  * The schedule of the redundant computation N°1: {c0[i,j]->c0[1,0,i,0,j,0]: 0<=i<N and 0<=j<N}
2949  * The schedule of the redundant computation N°2: {c0[i,j]->c0[2,0,i,0,j,0]: 0<=i<N and 0<=j<N}
2950  *
2951  * The schedule of c0 in this case would be three maps that map c0[i,j] to
2952  * the three different redundant computations in the time-processor domain:
2953  *
2954  * {c0[i,j]->c0[0,0,i,0,j,0]: 0<=i<N and 0<=j<N;
2955  * c0[i,j]->c0[1,0,i,0,j,0]: 0<=i<N and 0<=j<N;
2956  * c0[i,j]->c0[2,0,i,0,j,0]: 0<=i<N and 0<=j<N}
2957  *
2958  * The function set_schedule() overrides any other schedule set by the high level
2959  * scheduling functions. Currently the user has to choose between using the high
2960  * level scheduling functions or using this low level set_schedule function. The user
2961  * cannot mix the use of the two in the same program because they are not compatible.
2962  */

3081  * the iteration domain into full and partial iteration
3082  * domains for unrolling first and then it calls
3083  * tag_unroll_level().
3084  * tag_unroll_level() only tags a dimension to
3085  * be unrolled, it does not modify the tagged dimension.
3086  *
3087  * This function separates the iteration domain into two iteration
3088  * domains, a full iteration domain and a partial iteration domain.
3089  * The full iteration domain has an upper bound that is multiple of
3090  * \p fac while the other does not.
3091  * The full iteration domain is then split by \p fac and the inner loop
3092  * (which should have a constant extent equal to \p fac) is tagged as
3093  * a unrolled loop.
3094  *
3095  * Let us assume the following loop (a loop represents and iteration
3096  * domain)
3097  *
3098  * for (i=0; i<N; i++)
3099  * for (j=0; j<23; j++)
3100  * S0;
3101  *
3102  * To unroll the j loop with an unrolling factor of 4, one should call
3103  *
3104  * S0.unroll(j, 4);
3105  *
3106  * The loop (iteration domain) is first separated into the following
3107  * two loops
3108  *
3109  * for (int i=0; i<20; i++)
3110  * S0;
3111  *
3112  * for (int i=20; i<23; i++)
3113  * S0;
3114  *
3115  * The full loop is then split by 4
3116  *
3117  * for (int i1=0; i1<20/4; i1++)
3118  * for (int i2=0; i2<4; i2++)
3119  * S0;
3120  *
3121  * for (int i=20; i<23; i++)
3122  * S0;

3125  *
